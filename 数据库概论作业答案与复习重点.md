## 第一章 数据库基本概念

### 作业题目与答案

**1. 什么是数据独立性？它在数据管理技术发展历程中发挥怎样的作用？**

**答：**
- **数据独立性**是指当数据的结构（逻辑结构或存储结构）发生变化时，通过系统提供的映象（转换）功能，使应用程序不必改变。
- **分类**：
  - **物理独立性**：数据存储结构变化时，应用程序不变。
  - **逻辑独立性**：数据逻辑结构变化时，应用程序不变。
- **作用**：
  - 解决了文件系统中数据与程序紧密结合的问题。
  - 使应用程序员不必关心数据的物理存储细节。
  - 为数据库系统的推广和应用奠定了基础。

**2. 数据库系统是从哪些方面来提高数据独立性的？**

**答：**
- 通过**三级模式结构**（外模式、模式、内模式）和**两级映像**实现：
  - **外模式/模式映像**：保证逻辑独立性。模式改变时，只需修改映像，外模式不变。
  - **模式/内模式映像**：保证物理独立性。存储结构改变时，只需修改映像，模式不变。
- DBMS 作为数据管理的中间层，将数据定义、存取、管理等功能从应用程序中分离出来。

**3. 比较数据库系统和文件系统在数据管理方面的不同，列出至少四个不同之处。**

**答：**

| 比较项 | 文件系统 | 数据库系统 |
|--------|---------|-----------|
| 数据结构化 | 无结构或弱结构 | 结构化，统一存取 |
| 数据独立性 | 低，数据与程序紧密结合 | 高，三级模式两级映像 |
| 数据共享 | 差，信息孤岛 | 好，统一管理和共享 |
| 数据冗余 | 高，易不一致 | 低，一致性好 |
| 完整性 | 需程序自行维护 | DBMS 统一维护 |
| 并发控制 | 无 | 有 |
| 故障恢复 | 弱 | 有日志、备份、恢复机制 |
| 查询方式 | 每个查询需重新编程 | 用 SQL 等高级语言 |

**4. 不适合使用数据库而应该基于文件系统的应用的特点有哪些？列举两个例子。**

**答：**
- **特点**：
  - 数据量小、结构简单。
  - 访问模式简单，无需复杂查询。
  - 对并发、事务、恢复要求不高。
  - 需要高性能的流式读写（如日志、多媒体）。
- **例子**：
  - 操作系统的配置文件管理。
  - 简单的日志文件存储。
  - 图像/视频文件管理。

**5. 对不同模型（层次、网状、关系、面向对象模型）做一个表格来展示其模型结构、表达能力、数据独立性、操作简便性、适用场景等。**

**答：**

| 模型 | 结构 | 表达能力 | 数据独立性 | 操作简便性 | 适用场景 |
|------|------|---------|-----------|-----------|---------|
| 层次模型 | 树 | 只能表示1:N联系 | 较差 | 较复杂 | 早期系统、有层次关系的数据 |
| 网状模型 | 图 | 可表示M:N | 较差 | 复杂 | 复杂联系的数据 |
| 关系模型 | 二维表 | 基于关系代数，表达能力强 | 高 | 简单（SQL） | 通用数据库系统 |
| 面向对象模型 | 对象、类、继承 | 支持复杂对象和行为 | 较高 | 较复杂 | 工程数据、CAD/CAM |

**6. 如果采用多维数组作为数据模型，它比较适合管理哪些应用场景的数据？**

**答：**
- 多维数组具有**有序性**和**位置相关性**，适合管理：
  - 科学计算数据（如气象数据、遥感图像）。
  - 时间序列数据（如股票行情、传感器数据）。
  - 数据仓库中的 OLAP 立方体。
  - 图像/视频数据。
- 关系模型集合是无序的，不适合表达具有强空间/时间位置关系的数据。

**7. 画出数据库三层模式架构图，简单说明一下模式分层的好处。**

**答：**
- **架构图**：
  ```
        用户1  用户2  ...  用户n
         |     |          |
         +-----+----------+
                 |
              外模式（用户视图）
                 |
        外模式/模式映像
                 |
              模式（全局逻辑结构）
                 |
        模式/内模式映像
                 |
              内模式（物理存储结构）
                 |
               数据库
  ```
- **好处**：
  - 实现物理独立性和逻辑独立性。
  - 简化用户视图，隐藏复杂性。
  - 便于管理和维护。

**8. 搜索引擎的关键字查询和数据库查询有何不同？**

**答：**

| 比较项 | 搜索引擎关键字查询 | 数据库查询 |
|--------|------------------|-----------|
| 查询方式 | 基于关键词匹配 | 基于结构化查询语言（SQL） |
| 查询结果 | 相关性排序，可能不精确 | 精确匹配，结果确定 |
| 数据模型 | 非结构化/半结构化文本 | 结构化关系 |
| 一致性 | 允许近似和模糊 | 要求精确 |
| 更新 | 批量索引更新 | 实时事务更新 |

**9. DBMS 的主要数据控制功能应该有哪些？如果有一个淘宝达人想基于单机版数据库比如 Access 记录自己的购物流水，上面的哪些功能是他所不需要的？**

**答：**
- **DBMS 主要功能**：
  1. 数据定义（DDL）。
  2. 数据存取（DML）。
  3. 数据库运行管理（并发控制、安全性、完整性、恢复）。
  4. 数据组织存储和管理。
  5. 数据库建立和维护。
- **个人用户不需要的功能**：
  - 并发控制（单机单用户）。
  - 复杂的分布式恢复机制。
  - 角色/权限管理（可能简单即可）。
  - 高可用集群管理。

**10. DBA 应该履行哪些职能？相应的他又应该具备哪些技能？**

**答：**
- **职能**：
  - 参与数据库设计，确定模式、外模式、存储结构。
  - 数据整理和装入。
  - 定义完整性约束和安全规则。
  - 监督数据库运行，处理故障和恢复。
  - 性能优化和数据库重组。
- **技能**：
  - 数据库理论和设计能力。
  - 熟悉 DBMS 产品和 SQL。
  - 操作系统和存储知识。
  - 网络和分布式系统基础。
  - 问题诊断和性能调优能力。

**11. （选做）DB for AI / AI for DB 解释。**

**答：**
- **DB for AI**：为 AI 应用提供数据管理基础设施，如向量数据库、训练数据管理、数据版本控制。
- **AI for DB**：利用 AI/机器学习优化数据库内部组件，如学习型索引、学习型查询优化、自动调参、异常检测。

**12. 借鉴 DBMS 分层结构，启发大模型为你画出一张面向大模型的数据管理系统的架构图。**

**答：**（思路）可类比 DBMS 分层：
- 应用接口层（提示词管理、Agent 编排）。
- 语义层（知识表示、向量索引）。
- 存储管理层（向量数据库、文档库、关系库）。
- 资源调度层（模型调用、缓存、日志）。

### 复习重点（结合考纲）
- ⭐⭐⭐ **必考**：数据独立性（物理/逻辑）、三级模式两级映像。
- ⭐⭐ **常考**：DBMS 功能、DBA 职责、数据库系统 vs 文件系统。
- ⭐ 了解：不同数据模型对比。

## 第二章 ER 模型

### 作业题目与答题思路

**1. 分别列举聚集、弱实体、细化/泛化的实用例子（不要同讲义上的相同），说明优点。**

**答：**
- **聚集例子**：在电商系统中，"订单"与"商品"之间的"包含"联系本身可以作为一个实体参与"发货"联系。优点：可以表达联系上的联系，模型更自然。
- **弱实体例子**：房屋租赁系统中，"房间"依赖于"楼栋"存在，房间号在不同楼栋下可重复。优点：避免主码过于复杂，清晰表达依赖关系。
- **细化/泛化例子**：学校人事系统中，"教职工"泛化为"教师"和"行政人员"，教师有"职称"，行政人员有"岗位级别"。优点：减少冗余，表达共性属性和特有属性。

**2. 已知关系模式 R1(a1,a2,a3), R2(a3,a4,a1), R3(a5,a6), R4(a3,a5,a7), R5(a1,a3,a5,a8)，下划线为主码，重合属性为主外码关系，画出 ER 图。**

**答：**（文字描述）
- 实体 A（属性 a1, a2, a3），主码 a1。
- 实体 B（属性 a3, a4），主码 a3，外码 a3 参照 A？ 实际上 R2(a3,a4,a1) 主码是 a3，说明 B 主码 a3，与 A 有联系（通过 a1）。
- 更准确分析：
  - R1(a1,a2,a3)：实体 A，主码 a1，属性 a2, a3。
  - R2(a3,a4,a1)：实体 B，主码 a3，属性 a4，外码 a1 参照 A（1:N 或 1:1）。
  - R3(a5,a6)：实体 C，主码 a5，属性 a6。
  - R4(a3,a5,a7)：联系 AB-C 或实体 D，主码 (a3,a5)，说明 B 和 C 之间是 M:N 联系，a7 是联系属性。
  - R5(a1,a3,a5,a8)：三元联系 A-B-C，主码 (a1,a3,a5)，a8 是联系属性。
- 因此 ER 图包含：A、B、C 三个实体，以及 B-C 的 M:N 联系、A-B-C 的三元联系。

**3. NBA 球员转会表格 → ER 图。**

**答：**（思路）
- 实体：球员（PlayerID, Name, Position, ...）、球队（TeamID, TeamName, City, ...）。
- 联系：转会（Transfer），属性包括转会时间、转会费、原球队、新球队。
- 球员与球队之间是 M:N 联系（一名球员可效力多支球队，一支球队有多名球员）。

**4. 微信数据库相关表 → ER 图。**

**答：**（思路）
- 实体：信友（信友ID, 信友名, 昵称, 区域, 手机号）、群（群ID, 群名称, 群类型, 创建时间）。
- 联系：
  - 通讯录：信友与信友之间的 M:N 联系（认识方式、认识时间）。
  - 群成员：群与信友的 M:N 联系（加入时间、引介人、群内昵称），群主是 1:N 联系。
  - 帖子：群与信友之间的 1:N 联系（一个群有多条帖子，一个信友可发多条帖子）。
  - 短信：信友与信友之间的 M:N 联系。

**5. 论文评审数据库 → ER 图。**

**答：**（思路）
- 实体：论文、作者、审稿人、主题。
- 联系：
  - 论文与作者：M:N（含通讯作者标记）。
  - 论文与主题：N:1（一个论文一个主题）。
  - 审稿人与主题：M:N（关心主题）。
  - 主题与审稿人：1:1（主持人）。
  - 论文与审稿人：M:N（评审联系，属性：可读性、创新性、相关性分数，书面意见）。

**6. 航班/航线/机场/机组/飞机/飞行员 → ER 图。**

**答：**（思路）
- 实体：航班、航线、机场、机组、飞机、飞行员。
- 联系：
  - 航线连接两个机场（1:N，一个航线有多个航班）。
  - 航班属于一条航线（N:1）。
  - 航班由一架飞机执行（N:1）。
  - 机组由多名飞行员组成（1:N），执行多个航班（M:N）。
  - 航班与机组：N:1 或 M:N。

**7. 自选业务系统（知乎/小红书/拼多多）→ ER 图。**

**答：**（思路）
- 以知乎为例：
  - 实体：用户、问题、回答、评论、话题、关注。
  - 联系：用户提问（1:N）、用户回答（1:N）、用户评论（1:N）、回答属于问题（N:1）、问题属于话题（N:M）、用户关注话题/用户（M:N）。

### 复习重点（结合考纲）
- ⭐⭐⭐ **必考**：ER 图 → 关系模式转换规则（1:N、M:N、1:1、弱实体、聚集、概化）。
- ⭐⭐ **常考**：超码/候选码/主码/分辨符、弱实体。
- ⭐ 了解：特化/概化约束。

## 第三章 关系模型

### 作业题目与答案

**题目一：SPJ 数据库**

关系模式：
- S(SNO, SNAME, CITY)
- P(PNO, PNAME, COLOR, PRICE)
- J(JNO, JNAME, CITY)
- SPJ(SNO, PNO, JNO, QTY)

**1. 求向北京的工程供应了红色零件的供应商姓名**

**答：**
```
π_SNAME(σ_CITY='北京' ∧ COLOR='红色'(S ⋈ SPJ ⋈ P ⋈ J))
```
或更严格：
```
π_SNAME(σ_CITY='北京'(J) ⋈ SPJ ⋈ σ_COLOR='红色'(P) ⋈ S)
```

**2. 求同时向位于北京和天津的工程供应了零件的供应商的供应商名**

**答：**
```
π_SNAME(
    (π_SNO(σ_CITY='北京'(J) ⋈ SPJ) ∩ π_SNO(σ_CITY='天津'(J) ⋈ SPJ)) ⋈ S
)
```

**3. 求向和自己位于相同城市的工程供应零件的供应商的供应商号**

**答：**
```
π_SNO(σ_S.CITY=J.CITY(S ⋈ SPJ ⋈ J))
```

**4. 求只向和自己位于不同城市的工程供应零件的供应商的供应商号**

**答：**
```
π_SNO(SPJ ⋈ S) - π_SNO(σ_S.CITY=J.CITY(S ⋈ SPJ ⋈ J))
```

**5. 求向所有位于北京的工程都供应了零件的供应商的供应商号**

**答：**
```
π_SNO, JNO(SPJ) ÷ π_JNO(σ_CITY='北京'(J))
```

**6. 求价格最高的零件的零件号**

**答：**
```
π_PNO(P) - π_P1.PNO(σ_P1.PRICE < P2.PRICE(ρ_P1(P) × ρ_P2(P)))
```
或：
```
π_PNO(σ_PRICE = MAX(PRICE)(P))
```

**题目二：选课表 SC(sno, cno, grade)**

**1. 求恰好选修了 c1 和 c2 课程的学生**

**答：**
```
π_sno(σ_cno='c1'(SC) ⋈ σ_cno='c2'(SC)) - π_sno(σ_cno≠'c1' ∧ cno≠'c2'(SC))
```
（或理解为：只选 c1 和 c2，没选其他课程）

**2. 求选修了所有 s1 同学所修课程的学生**

**答：**
```
π_sno, cno(SC) ÷ π_cno(σ_sno='s1'(SC))
```

**3. 求其选修课程被 s1 同学所修课程完全包含的学生**

**答：**
```
π_sno(SC) - π_sno((π_sno(SC) × π_cno(σ_sno='s1'(SC))) - π_sno, cno(SC))
```

**4. 求和 s1 同学所修课程完全不同的学生**

**答：**
```
π_sno(SC) - π_sno(SC ⋈ (π_cno(σ_sno='s1'(SC))))
```

**5. 求和 s1 同学所修课程完全相同的学生**

**答：**
```
π_sno(SC ÷ π_cno(σ_sno='s1'(SC))) ∩ π_sno(σ_cno∈π_cno(σ_sno='s1'(SC))(SC))
```
（更准确：该学生所选课程集合 = s1 所选课程集合）

**6. 求所修课程完全不同的学生对的**

**答：**（思路）
```
{(sA, sB) | π_cno(σ_sno=sA(SC)) ∩ π_cno(σ_sno=sB(SC)) = ∅}
```

**7. 求所修课程完全相同的学生对的**

**答：**（思路）
```
{(sA, sB) | π_cno(σ_sno=sA(SC)) = π_cno(σ_sno=sB(SC))}
```

**题目三：**

**1. 对于 R(A,B)，用关系代数检验 A 是否取值唯一。**

**答：**
```
π_A(R) 的元组数 = R 的元组数，且 π_A(R) 中无重复
```
用关系代数表达：
```
π_A(R) ⋈ π_A(R) 中若存在 A 相同但 B 不同的元组，则不唯一
```
更严格：
```
π_A(R) 的基数 = R 的基数 ⟺ A 取值唯一
```

**2. 对于 R(A,B,C)，检验 A 是否取值唯一。**

**答：**
```
π_A(R) 的基数 = R 的基数 ⟺ A 取值唯一
```
等价于：不存在两个元组在 A 上相同但在 B 或 C 上不同。

**3. 用基本关系代数操作表达 R(A,B) ⟕ S(B,C)。**

**答：**
```
R ⟕ S = (R ⋈ S) ∪ ((R - π_R(R ⋈ S)) × {NULL})
```
其中 {NULL} 表示用空值扩展 schema (B,C)。
更完整：
```
(R ⋈ S) ∪ π_A,B,NULL→C(R - π_A,B(R ⋈ S))
```

**题目四：用元组关系演算和域关系演算**

**1. 求同时选修了 c1 和 c2 课程的学生**

**答：**
- 元组关系演算：
  ```
  { t | ∃s1∈SC ∃s2∈SC (t[sno] = s1[sno] ∧ s1[sno] = s2[sno] ∧ s1[cno]='c1' ∧ s2[cno]='c2') }
  ```
- 域关系演算：
  ```
  { s | ∃g1 ∃g2 (SC(s, 'c1', g1) ∧ SC(s, 'c2', g2)) }
  ```

**2. 求选修 c1 课程成绩比 s1 同学的该门课程成绩高的学生**

**答：**
- 元组关系演算：
  ```
  { t | ∃s1∈SC (t[sno] = s1[sno] ∧ s1[cno]='c1' ∧ ∃s2∈SC (s2[sno]='s1' ∧ s2[cno]='c1' ∧ s1[grade] > s2[grade])) }
  ```
- 域关系演算：
  ```
  { s | ∃g1 (SC(s, 'c1', g1) ∧ ∃g2 (SC('s1', 'c1', g2) ∧ g1 > g2)) }
  ```

## 第七章 关系规范化

### 作业题目与详细解答

**题目一：R(ABCDE), F = { AB→C, B→D, CD→E, CE→B, AC→B }**

**1. 候选码**

**解：**
- 求属性分类：
  - 左部属性：A（只出现在左边）
  - 右部属性：D, E（只出现在右边）
  - 双向属性：B, C
- 计算 A⁺：
  - A⁺ = {A}
  - 无法推出其他属性，需要加入 B 或 C。
- 计算 AB⁺：
  - AB → C, B → D, CD → E
  - AB⁺ = {A, B, C, D, E} = U
  - 所以 AB 是候选码。
- 检查 AC：
  - AC → B, B → D, CD → E
  - AC⁺ = {A, C, B, D, E} = U
  - 所以 AC 也是候选码。
- **候选码：AB, AC**。

**2. 判断范式级别**

**解：**
- 主属性：A, B, C
- 非主属性：D, E
- 检查 2NF：是否存在非主属性对候选码的部分依赖。
  - AB → D？B → D，B 是 AB 的真子集，所以 D 部分依赖于 AB。
  - 因此 **R 不满足 2NF**。
- **结论：R ∈ 1NF**。

**3. 分别给出保持无损连接和函数依赖的分解**

**解：**
- **保持函数依赖的 3NF 分解**：
  1. 求最小覆盖 Fmin：
     - 右部单属性化：F 已经是。
     - 检查冗余：
       - AB→C：去掉后 AB⁺ 是否还包含 C？用 F - {AB→C}，AB⁺ = {A,B,D}，不包含 C，不冗余。
       - B→D：不冗余。
       - CD→E：不冗余。
       - CE→B：不冗余。
       - AC→B：AC→B 可由 AC→C 和 ... 实际上 AC→B 不是冗余。
     - 检查左部既约：
       - AB→C：A⁺={A}, B⁺={B,D}，都不能推出 C，所以 AB 不可约。
       - AC→B：A⁺={A}, C⁺={C}，都不能推出 B，所以 AC 不可约。
     - Fmin = F。
  2. 每个函数依赖组成一个关系：
     - R1(ABC) 对应 AB→C
     - R2(BD) 对应 B→D
     - R3(CDE) 对应 CD→E
     - R4(BCE) 对应 CE→B
     - R5(ABC) 对应 AC→B（与 R1 重复）
  3. 检查是否包含候选码：R1 包含候选码 AB，满足。
  4. 去掉被包含的：R5 与 R1 重复，去掉。
  5. **最终分解：ρ = {R1(ABC), R2(BD), R3(CDE), R4(BCE)}**。

- **保持无损连接的 BCNF 分解**：
  - R 不满足 BCNF，因为 B→D 中 B 不是超码。
  - 分解：R1(BD), R2(ABCE)。
  - 检查 R2(ABCE)：
    - F2 = {AB→C, CD→E, CE→B, AC→B} 在 ABCE 上的投影。
    - 候选码仍为 AB 或 AC。
    - AB→C 中 AB 是超码，满足 BCNF。
    - 但 CE→B 中 CE 不是超码（CE⁺ = {C,E,B}，不包含 A），所以 R2 不满足 BCNF。
  - 继续分解 R2：用 CE→B，分解为 R3(BCE), R4(ACE)。
  - 检查 R3(BCE)：CE 是候选码，CE→B 中 CE 是超码；B→? 没有，满足 BCNF。
  - 检查 R4(ACE)：AC 是候选码，AC→? 在 R4 上没有非平凡依赖，满足 BCNF。
  - **最终分解：ρ = {R1(BD), R3(BCE), R4(ACE)}**。

**题目二：R(ABCDE), F = { AB→D, AC→E, BC→D, D→A, E→B }，求 F 在 S(ABCD) 上的投影。**

**解：**
- 投影 π_S(F) = { X→Y | X→Y ∈ F⁺ 且 XY ⊆ ABCD }。
- 先求 F⁺ 中涉及 ABCD 的依赖：
  - 直接：AB→D, BC→D, D→A
  - 推导：
    - D→A, AB→D ⟹ DB→D（平凡）
    - AB→D, D→A ⟹ AB→A（平凡）
    - 由 D→A 和 BC→D 可得 BC→A
    - 由 BC→A 和 BC→D 可得 BCD→AD 等
- **投影结果**：{ AB→D, BC→D, D→A, BC→A }。

**题目三：R(BCDFGH), F = { BG→CD, G→F, CD→GH, C→FG, F→D }，给出同时保持函数依赖和无损的 3NF 分解。**

**解：**
- **步骤 1：求候选码**
  - 左部属性：B
  - 右部属性：D, F, G, H
  - 双向属性：C
  - 计算 B⁺：{B}
  - 计算 BC⁺：C→F, F→D, BC→? 无法直接推出 G, H。
  - 计算 BG⁺：BG→C, G→F, CD→G, H... 实际上 BG→CD，所以 BG⁺ = {B,G,C,D}；CD→GH，所以 BG⁺ = {B,G,C,D,G,H} = U。
  - 因此 **BG 是候选码**。
  - 检查是否还有其他候选码：从 B 出发，必须包含 G 才能得到 C,D,H，所以 BG 唯一。

- **步骤 2：求最小覆盖**
  - F = { BG→C, BG→D, G→F, CD→G, CD→H, C→F, C→G, F→D }
  - 右部单属性化后。
  - 检查冗余：
    - BG→D：能否从 F - {BG→D} 推出 BG→D？
      - BG→C, C→G（已有），BG→C 和 CD→G 等。
      - BG→C, C→F, F→D，所以 BG→D 可推导，冗余。
    - C→G：CD→G，所以 C→G 可由 CD→G 和 C→C 推导？不能直接用，因为需要 D。
      - 实际上 C⁺ 在 F 中 = {C, F, G, D}，所以 C→G 可由 C→F, F→D, 但 D→? 不能得到 G。
      - 但 CD→G，所以 C→G 不是冗余（去掉后 C⁺ 不再包含 G）。
  - 简化后最小覆盖约为：
    - Fmin = { BG→C, G→F, CD→G, CD→H, C→G, C→F, F→D }
    - 进一步 C→G, G→F, F→D 可得 C→F, C→D，所以 C→F 冗余。
  - 最终 Fmin ≈ { BG→C, G→F, CD→G, CD→H, C→G, F→D }

- **步骤 3：3NF 合成**
  - 每个函数依赖组成关系：
    - R1(BGC) 对应 BG→C
    - R2(GF) 对应 G→F
    - R3(CDG) 对应 CD→G
    - R4(CDH) 对应 CD→H
    - R5(CG) 对应 C→G
    - R6(FD) 对应 F→D
  - 检查候选码 BG 是否被包含：R1(BGC) 包含 BG，满足。
  - 去掉被包含的：R5(CG) 被 R3(CDG) 包含（CG ⊆ CDG），去掉。
  - **最终分解**：ρ = {R1(BGC), R2(GF), R3(CDG), R4(CDH), R6(FD)}。
  - 验证无损：由于包含候选码 BG，且 R1 包含 BG，所以分解保持无损。

**题目四：R(ABCD)，F = { A→BCD }，多值依赖 B→→C，判断 R 的范式级别。**

**解：**
- 函数依赖 A→BCD，A 是超码，所以 R ∈ BCNF。
- 但存在非平凡多值依赖 B→→C，且 B 不是超码，所以 R 不满足 4NF。
- **结论：R ∈ BCNF 但不满足 4NF**。

**题目五：给出判断 r(ABC) 上多值依赖 A→→B 是否成立的关系代数和 SQL 语句。**

**解：**
- **多值依赖 A→→B 成立条件**：对于关系中每一个 A 值，对应的 B 值集合与 C 值集合独立。
- **关系代数判断**：
  ```
  π_AB(r) ⋈ π_AC(r) = r
  ```
  若等式成立，则 A→→B 成立。
- **SQL 判断**：
  ```sql
  -- 若以下查询结果为空，则 A→→B 成立
  SELECT * FROM r
  WHERE NOT EXISTS (
      SELECT * FROM r r2
      WHERE r.A = r2.A
      AND EXISTS (
          SELECT * FROM r r3
          WHERE r3.A = r.A
          AND r3.B = r.B
          AND r3.C = r2.C
      )
  );
  ```
  更简洁：
  ```sql
  SELECT COUNT(*) FROM (
      (SELECT A, B FROM r) CROSS JOIN (SELECT A, C FROM r)
      EXCEPT
      SELECT A, B, C FROM r
  ) t;
  -- 结果为 0 则 A→→B 成立
  ```

**题目六：求 F = { ABCD→E, E→D, A→B, AC→D } 的最小覆盖。**

**解：**
- 步骤 1：右部单属性化
  - F = { ABCD→E, E→D, A→B, AC→D }
- 步骤 2：检查冗余
  - ABCD→E：A⁺={A,B}, B⁺={B}, C⁺={C}, D⁺={D}，不能推出 E，所以 ABCD 左部不可约；依赖本身不冗余。
  - E→D：不冗余。
  - A→B：不冗余。
  - AC→D：A→B，AC→? 能否去掉 C？A⁺={A,B}，不能推出 D，所以 C 不可去掉；AC→D 不冗余。
- 步骤 3：检查左部既约
  - ABCD→E：去掉任一属性后都不能推出 E，所以不可约。
- **最小覆盖 Fmin = { ABCD→E, E→D, A→B, AC→D }**。

**题目七：R(ABCDEG), F = { A→G, C→AD, B→AE, CG→B }，用 ER 图表示。**

**解：**
- 步骤 1：求最小覆盖
  - F = { A→G, C→A, C→D, B→A, B→E, C→B, G→B }
  - 检查冗余：
    - C→A：C→B, B→A，所以 C→A 可由 C→B 和 B→A 推导，冗余。
    - C→B, B→A, B→E，所以 C→A 和 C→E 可推导。
    - CG→B：C→B，所以 CG→B 冗余。
  - Fmin ≈ { A→G, C→D, B→A, B→E, C→B }
- 步骤 2：3NF 合成
  - R1(AG) 对应 A→G
  - R2(CD) 对应 C→D
  - R3(BAE) 对应 B→A, B→E
  - R4(CB) 对应 C→B
  - 候选码？C⁺ = {C, D, B, A, E, G} = U，所以 C 是候选码。
  - R2(CD) 包含 C，满足。
  - 去掉被包含：无。
  - **分解**：{AG, CD, BAE, CB}。
- 步骤 3：画 ER 图
  - 实体：C（主码 C，属性 D, B），B（主码 B，属性 A, E），A（主码 A，属性 G）。
  - 联系：C 与 B 之间的 1:1 或 N:1 联系（C→B），B 与 A 之间的 N:1 联系（B→A），A 与 G 之间的 N:1 联系（A→G）。

```sql

FROM Student stu
```

## 第八章 事务

### 作业题目与答案

**1. 当一个事务中的某条 SQL 语句遇到运行错误时，该事务该如何进行善后处理？**

**答：**
- 可恢复错误（如违反约束）：该语句执行失败，事务是否继续取决于 DBMS 实现（MySQL 默认不会自动回滚整个事务）。
- 严重错误（如死锁、连接断开）：整个事务回滚。
- 应用程序应捕获错误，根据业务逻辑决定 COMMIT 或 ROLLBACK。

**2. 阐述事务一致性和数据完整性的对比。**

**答：**
- **事务一致性（Consistency）**：ACID 之一，事务执行前后数据库都处于一致状态。
- **数据完整性（Integrity）**：数据必须满足的静态规则，包括实体完整性、参照完整性、用户定义完整性。
- **关系**：事务一致性是过程正确性，数据完整性是目标规则；事务一致性通过原子性、隔离性、持久性来保证不破坏完整性。

**3. 阐述一致性和原子性的对比。**

**答：**
- **原子性（Atomicity）**：事务要么全做，要么全不做。
- **一致性（Consistency）**：事务执行结果正确，满足所有约束。
- **关系**：原子性是手段，一致性是目标；原子性防止部分更新导致不一致。

**4. n 个事务，分别有 m₁, m₂, ..., mₙ 条指令，总指令数 M = Σmᵢ，可能的并发调度有多少个？**

**答：**
```
M! / (m₁! × m₂! × ... × mₙ!)
```
解释：M 条指令的全排列中，同一事务内部指令相对顺序不变。

**5. 列举一些必须要重复读的业务场景。**

**答：**
- 银行转账：读取余额后基于该余额操作。
- 库存管理：读取库存后判断是否可下单。
- 机票/火车票预订：查询余座后下订单。
- 财务报表生成：多次读取同一数据范围。

**6. 无级联调度会发生哪些不一致现象？**

**答：**
无级联调度只能避免读脏数据，仍可能发生：
- 丢失修改
- 不可重复读
- 幻象读

**7. 要防止发生丢失修改，事务至少要设置在哪个隔离性级别上？**

**答：**
- 理论上至少 **Repeatable Read**（通过长锁或 MVCC first-committer-wins）。
- 实际上 Serializable 最安全。

**8. 要防止发生幻象，事务至少要设置在哪个隔离性级别上？**

**答：**
- SQL 标准：至少 **Serializable**。
- MySQL InnoDB：Repeatable Read 通过 Next-Key Lock 可防止幻象读。

**9. 为什么在快照隔离中，读写操作不会产生冲突？**

**答：**
- 快照隔离基于 MVCC，读操作读取事务开始时的快照版本，写操作创建新版本。
- 读写操作的是不同版本的数据，因此不直接冲突（读不阻塞写，写不阻塞读）。
- 只有写-写冲突（first-committer-wins）会导致回滚。

**10. 什么是快照隔离中的写偏斜现象？要避免写偏斜，事务至少要设置在哪个隔离性级别上？**

**答：**
- **写偏斜（Write Skew）**：两个事务分别读取重叠数据集，各自修改不同部分，各自通过约束检查，但合起来违反全局约束。
- 经典例子：医生值班系统，两个事务分别让两位医生下班，结果无人值班。
- 避免：至少 **Serializable** 隔离级别，或使用 SELECT ... FOR UPDATE。

**11. 语句级快照隔离 RCSI 更接近 RC，事务级快照隔离 SI 更接近 RR，对吗？它们彼此的差别是什么？**

**答：**
- 对。
- **RCSI**：每条语句获取新快照，避免脏读，但不可重复读。
- **SI**：事务开始时获取快照，整个事务使用同一快照，保证可重复读。
- 差别：RCSI 每次读都更新视图，SI 事务级视图；SI 可能出现写偏斜，RCSI 不会。

**12. 什么是 MySQL 的当前读和快照读？**

**答：**
- **快照读**：普通 SELECT，不加锁，读取快照版本（MVCC）。
- **当前读**：读取最新已提交版本，加锁，包括 SELECT ... FOR UPDATE、SELECT ... LOCK IN SHARE MODE、UPDATE、DELETE、INSERT。

**13. 构造满足如下复合条件的调度：**
- 无级联 + 非冲突可串行化
- 级联 + 冲突可串行化
- 无级联 + 非视图可串行化
- 级联 + 视图可串行化

**答：**
- **无级联 + 非冲突可串行化**：两个事务读写不同数据项，但形成循环依赖（视图可串行化但不冲突可串行化）。
- **级联 + 冲突可串行化**：T1 写 A（未提交），T2 读 A，冲突图无环但 T2 读了未提交数据。
- **无级联 + 非视图可串行化**：两个事务都提交，但最终写结果与任何串行调度不同。
- **级联 + 视图可串行化**：T1 写 A（未提交），T2 读 A 并写 B，T1 提交；视图等价于 T1→T2 串行。

### 复习重点（结合考纲）
- ⭐⭐⭐ **必考**：ACID、四种数据不一致性、四个隔离级别。
- ⭐⭐⭐ **必考**：冲突可串行化判定（优先图）、视图可串行化。
- ⭐⭐ **常考**：快照隔离、写偏斜、当前读/快照读。

## 第九章 并发控制

### 作业题目与答案

**1. 2PL 可以保证调度是可串行化的吗？可以保证无级联吗？可以避免不可重复读吗？哪个隔离级别肯定满足 2PL？**

**答：**
- 2PL 保证 **冲突可串行化**。
- 基本 2PL **不能保证**无级联；严格 2PL（X 锁长锁到事务结束）可以保证无级联。
- 2PL **不能保证**避免不可重复读，需要对读也加锁。
- **Serializable** 隔离级别肯定满足 2PL（或严格 2PL）。

**2. 升级锁和重新申请锁有区别吗？在哪个隔离级别下会出现锁转换？避免转换死锁的关键是什么？区分只读和先读后写。持有 U 锁可以对数据执行什么操作？U 和 S 相容吗？U 和 U 相容吗？**

**答：**
- **锁升级**：同一事务中将 S 锁提升为 X 锁（不释放再获取）。
- **重新申请锁**：先释放已有锁再申请新锁，可能导致其他事务抢占。
- 锁转换常见于 **Repeatable Read / Serializable**。
- 避免转换死锁：事务开始时直接申请足够高级别的锁（如 U 锁或 X 锁）。
- **只读**：S 锁；**先读后写**：U 锁（可升级为 X 锁）。
- U 锁与 S 锁相容，U 锁与 U 锁相容，U 锁与 X 锁不相容。

**3. 如何权衡封锁粒度大小的利弊？为什么需要支持多粒度锁？如何避免不同粒度的锁之间的冲突？**

**答：**
- **粗粒度锁**（表级）：开销小、并发度低，适合大事务/批处理。
- **细粒度锁**（行级）：并发度高、开销大，适合 OLTP。
- **多粒度锁**：不同事务访问范围不同，需要灵活选择粒度。
- **避免冲突**：使用意向锁（IS、IX、SIX），在加细粒度锁前先对父节点加意向锁。

**4. 和 IS 相容的锁有哪些？和 IX 相容的锁有哪些？SIX 适合什么应用场合？它和哪些锁相容？**

**答：**
- 与 IS 相容：IS、IX、S、SIX。
- 与 IX 相容：IS、IX。
- SIX：事务要读整个表并修改其中部分行；只与 IS 相容。

**5. 死锁发生的条件有哪些？从预防死锁的角度，列举一些措施。**

**答：**
- 死锁四条件：互斥、占有等待、不可剥夺、循环等待。
- 预防措施：
  - 一次封锁所有资源（破坏占有等待）。
  - 按固定顺序加锁（破坏循环等待）。
  - 超时回滚（破坏不可剥夺）。
  - wait-die / wound-wait（按时间戳优先级）。

**6. 讨论事务隔离性级别、事务粒度、封锁粒度和死锁发生几率的关系。**

**答：**
- 隔离级别越高，锁越多，死锁几率越高。
- 事务越长，持有锁时间越长，死锁几率越高。
- 封锁粒度越细，锁数量越多，死锁几率越高。
- 综合：高隔离 + 长事务 + 细粒度 = 死锁风险最大。

**7. 讨论影响事务吞吐率的措施。**

**答：**
- 降低隔离级别。
- 减小事务粒度。
- 使用行级锁。
- 使用 MVCC。
- 索引优化。
- 连接池、读写分离、分库分表。
- 消息队列异步化。

**8. 列举可以提高事务吞吐率的措施。**

**答：**（同第 7 题）降低隔离级别、减小事务、行级锁、乐观锁、MVCC、索引优化、连接池、读写分离、分库分表、消息队列、批量提交、合理超时。

**9. 什么是活锁？如何避免活锁？**

**答：**
- **活锁**：事务一直等待某锁，但总是被其他事务抢先获取。
- 避免：先来先服务（FCFS）队列、公平调度、随机退避。

**10. 在时间戳协议中，如何确定一个读写操作是正常的？什么是跳过的写？提交位的作用是什么？**

**答：**
- 每个数据项 Q 维护 WT(Q)、RT(Q)。
- **读操作 R_T(Q)**：若 TS(T) < WT(Q)，则过晚读，T 回滚；否则正常。
- **写操作 W_T(Q)**：若 TS(T) < RT(Q) 或 TS(T) < WT(Q)，则过晚写，T 回滚（Thomas 写规则下可跳过而不回滚）。
- **跳过的写**：写操作因时间戳过时被忽略（Thomas 写规则）。
- **提交位**：标记事务是否已提交，防止读到未提交数据，避免级联回滚。

**11. 在有效性检查协议中，读写集合与已确认事务的写集合如何判定？**

**答：**
- 事务分读阶段、验证阶段、写阶段。
- **向后验证**：对更早确认的事务 T_valid，检查 RS(T) ∩ WS(T_valid) = ∅；若 T_valid 与 T 并发，还需检查 WS(T) ∩ WS(T_valid) = ∅。
- 若冲突，T 回滚。

**12. 在 MVCC 中，读视图结构是怎样的？可见性算法思想是怎样的？RC 和 RR 的读视图区别是什么？**

**答：**
- **读视图结构**：m_ids（活跃事务列表）、min_trx_id、max_trx_id、creator_trx_id。
- **可见性算法**：
  - trx_id == creator_trx_id：可见。
  - trx_id < min_trx_id：可见。
  - trx_id >= max_trx_id：不可见。
  - min_trx_id <= trx_id < max_trx_id：trx_id 在 m_ids 中则不可见，否则可见。
- **RC 与 RR 区别**：RC 每条语句新建读视图；RR 事务开始时创建并复用。

- ⭐⭐⭐ **必考**：死锁检测与预防、时间戳协议、MVCC 读视图与可见性。
- ⭐⭐ **常考**：多粒度锁、意向锁、有效性检查协议。

## 第十章 恢复控制

### 作业题目与答案

**1. 恢复的目标有哪些？恢复的措施有哪些？故障类型有哪些？备份的类型有哪些？**

**答：**
- **恢复目标**：保证原子性、持久性；将数据库恢复到一致状态；最小化恢复时间和数据丢失。
- **恢复措施**：日志、检查点、备份、镜像。
- **故障类型**：事务故障、系统故障、介质故障、灾难。
- **备份类型**：完全备份、增量备份、差异备份、日志备份；热备、冷备、温备。

**2. 什么是影子分页技术？**

**答：**
- 维护当前页表和影子页表。
- 事务修改在新页（影子页）上进行，更新当前页表指向新页。
- 提交时写入当前页表；回滚时恢复影子页表。
- 优点：回滚快、不需要日志；缺点：碎片化、不适合并发。

**3. 日志基本的恢复操作是什么？什么是 WAL 原则？**

**答：**
- **基本操作**：Redo（重做已提交但未落盘修改）、Undo（撤销未提交修改）。
- **WAL（Write-Ahead Logging）**：数据页写入磁盘前，相关日志必须先写入磁盘；COMMIT 日志在修改落盘前刷盘。

**4. 日志类型有哪些？比较维度有哪些？**

**答：**
- 按格式：物理日志、逻辑日志、生理日志。
- 按操作：Undo 日志、Redo 日志、Undo/Redo 日志、CLR（补偿日志）。
- 比较维度：日志大小、恢复速度、并发恢复支持、回滚复杂度、与物理存储独立性。

**5. 各种故障类型的恢复过程是怎样的？**

**答：**
- **事务故障**：反向扫描日志，Undo 该事务所有操作。
- **系统故障**：正向扫描 Redo 已提交事务；反向扫描 Undo 未提交事务。
- **介质故障**：从备份恢复，应用增量/差异备份，再应用日志 Redo，最后 Undo 未提交事务。

**6. 什么是检查点？它的作用是什么？**

**答：**
- **检查点**：将脏页刷新到磁盘，并在日志中记录检查点记录的事件。
- **作用**：缩短恢复时间；减少 Undo/Redo 工作量；管理日志空间。
- **模糊检查点**：不强制刷新所有脏页，只记录 DPT 和 ATT。

**7. 针对事务的脏写是否可以落盘和提交写是否必须落盘所形成的各种缓冲区管理策略，对日志类型的需求是怎样的？**

**答：**
- **No-Steal + Force**：不需要 Undo/Redo 日志。
- **No-Steal + No-Force**：需要 Redo 日志。
- **Steal + Force**：需要 Undo 日志。
- **Steal + No-Force**（最常用）：需要 Undo + Redo 日志。
- 性能：Steal + No-Force 最好，但恢复最复杂。

**8. ARIES 算法相应的页结构、ATT、DPT 是怎样的？什么是补偿日志？**

**答：**
- **页结构**：每个数据页包含 Page_LSN（最近一次修改该页的日志 LSN）。
- **ATT（活跃事务表）**：记录未提交事务，含 TransID、Status、Last_LSN、Undo_Next_LSN。
- **DPT（脏页表）**：记录脏页，含 Page_ID、RecLSN（使页面变脏的第一条日志 LSN）。
- **CLR（补偿日志记录）**：记录 Undo 操作本身，包含 Undo_Next_LSN 指向下一个需 Undo 的日志；保证回滚幂等，避免重复回滚。

**9. ARIES 算法的三个阶段是怎样的？各个阶段的处理起点是怎样的？**

**答：**
- **分析阶段**：从最近检查点开始正向扫描，重建 ATT 和 DPT。
- **重做阶段**：从 DPT 中最小 RecLSN 开始正向扫描，Redo 需要重做的操作。
- **撤销阶段**：从需要 Undo 的事务的最近日志开始反向扫描，Undo 并写 CLR。

**10. 扩展讨论：agent harness 工程和数据库恢复控制的技术契合点。**

**答：**
- Agent 工作流可类比事务模型（ACID）。
- Saga 模式/补偿机制对应 Undo/CLR。
- 检查点对应 Agent 状态保存与断点续传。
- WAL 思想可用于 Agent 操作审计与回滚。
- MVCC 可用于多 Agent 并发控制。

### 复习重点（结合考纲）
- ⭐⭐⭐ **必考**：WAL 原则、Redo/Undo、检查点。
- ⭐⭐⭐ **必考**：ARIES 三阶段、PageLSN、DPT、ATT、CLR。
- ⭐⭐ **常考**：故障类型、备份类型、Steal/No-Force 策略。

## 第十一章 数据库存储

### 作业题目与答案

**1. 磁盘顺序读和随机读的性能差异对数据库设计有哪些方面的影响？**

**答：**
- 使用 B+ 树索引和叶子节点链表，支持顺序扫描。
- 预读机制（Read-Ahead）。
- 缓冲池管理（LRU、局部性）。
- 聚簇索引减少随机 I/O。
- WAL 将随机写转化为顺序写。
- LSM 树将随机写转化为顺序写。
- 列存储适合分析型顺序扫描。

**2. 各个 RAID 级别设计机制的差异？数据库该如何选择合适的 RAID 级别？**

**答：**
- **RAID 0**：条带化，无冗余，性能高。
- **RAID 1**：镜像，高可靠，读性能好。
- **RAID 5**：条带 + 分布式奇偶校验，容单盘故障，读多写少。
- **RAID 6**：双奇偶校验，容双盘故障。
- **RAID 10**：镜像 + 条带，性能与可靠性兼顾。
- 选择：OLTP 用 RAID 10；OLAP 用 RAID 5/6；日志用 RAID 1/10。

**3. MySQL 和 SQL Server 的缓冲区管理策略是如何具体实现的？MySQL 的 change buffer 和 double write buffer 的作用是什么？**

**答：**
- **MySQL InnoDB Buffer Pool**：缓存数据页、索引页；改良 LRU（Young/Old 子列表）；多缓冲池实例；Page Cleaner 异步刷脏页。
- **SQL Server Buffer Pool**：Clock 算法近似 LRU；Lazy Writer 和 Checkpoint 刷脏页。
- **Change Buffer**：缓存对二级索引页的变更，减少随机 I/O。
- **Double Write Buffer**：解决部分页写入问题，先顺序写入双写缓冲，再写入数据文件。

**4. 数据页结构和数据行结构是怎样？**

**答：**
- **InnoDB 数据页（16KB）**：File Header → Page Header → Infimum/Supremum → User Records → Free Space → Page Directory → File Trailer。
- **COMPACT 行格式**：变长字段长度列表 → NULL 标志位 → 记录头信息 → Row ID → Transaction ID → Roll Pointer → 列数据。

**5. B 树和 B+ 树的区别是什么？**

**答：**
- B 树数据存储在所有节点；B+ 树数据只在叶子节点，内部节点只存键。
- B+ 树叶节点通过链表连接，范围查询高效。
- B+ 树内部节点扇出更大，树更矮，I/O 更少。
- B+ 树是现代数据库索引标准。

**6. 位图索引和位片索引是如何构造的？它们是如何支持查询的？**

**答：**
- **位图索引**：每个键值对应一个位图向量，长度为行数；等值查询直接取位图，复合条件用位运算。
- **位片索引**：将数值按二进制位切片，每个位片一个位图；支持范围查询和聚合（SUM/AVG）。

**7. 按列存储的优点是什么？如何理解 DuckDB 的向量化执行以及零拷贝技术？**

**答：**
- **列存储优点**：减少 I/O、高压缩率、CPU 缓存友好、适合 SIMD、延迟物化。
- **向量化执行**：一次处理一批数据（向量），减少函数调用开销，利用 SIMD。
- **零拷贝**：与 Apache Arrow 内存格式集成，数据在系统间共享时不复制；谓词下推、分区裁剪减少数据读取。

**8. 列举几种数据库里面的多维索引类型。**

**答：**
- R 树、R* 树、R+ 树
- KD 树
- 四叉树、八叉树
- 网格索引
- UB 树（Z-order + B+树）
- 位图联合索引
- iDistance

**9. LSM 树的思想是什么？相比 B+ 树的优点是什么？为什么它需要配合 Bloom filter？**

**答：**
- **LSM 树思想**：写入先存内存 MemTable，满后刷入磁盘 SSTable，后台合并（Compaction）；将随机写转化为顺序写。
- **优点**：写性能高、压缩率高、磁盘友好。
- **Bloom filter**：快速判断键是否在某个 SSTable 中，减少不必要的磁盘读取；无假阴性。

- ⭐⭐ **常考**：RAID 1/5/10、B+树 vs LSM 树、位图索引、列存储。
- ⭐⭐ **常考**：页结构、行结构、Change Buffer、Double Write Buffer。
- ⭐ 了解：多维索引、向量化执行、零拷贝。

## 第十二章 查询处理

### 作业题目与答案

**1. 数据库查询处理所包含的完整步骤有哪些？**

**答：**
1. 查询解析（词法/语法分析，语义检查）。
2. 查询转换/重写（视图展开、子查询去关联、谓词化简）。
3. 查询优化（RBO 启发式规则 + CBO 代价估算）。
4. 查询执行（物理算子、火山/向量化/编译执行）。
5. 结果返回。

**2. 理解两种基于规则的和基于代价的查询优化策略。**

**答：**
- **RBO（Rule-Based Optimization）**：预定义启发式规则，不依赖统计信息；如选择下推、投影下推、子查询去关联、视图合并。
- **CBO（Cost-Based Optimization）**：基于统计信息估算代价，选择最小代价计划；统计信息包括行数、直方图、NDV、NULL 比例等。
- 实际数据库通常先 RBO 改写，再 CBO 选择物理计划。

**3. 看懂各种访问方式的具体执行计划。**

**答：**
- **全表扫描 Seq Scan/Table Scan**：顺序读取所有页。
- **聚簇索引扫描 Clustered Index Scan**：按聚簇索引顺序扫描。
- **索引查找 Index Seek/Unique Scan**：通过 B+ 树定位行。
- **二级索引查找 + 回表**：先查二级索引，再回主键索引取数据。
- **索引范围扫描 Index Range Scan**：范围条件顺序扫描叶子节点。
- **位图扫描 Bitmap Scan**：构建位图后批量回表。
- **索引条件下推 ICP**：将 WHERE 条件下推到存储引擎层。

**4. 理解各种算子的物理实现策略，主要是各种连接实现策略，它们的适用场景。**

**答：**

| 连接算法 | 策略 | 适用场景 |
|---------|------|---------|
| 嵌套循环连接 NLJ | 外表逐行扫描内表 | 外表小、内表大 |
| 索引嵌套循环连接 | 内表有索引，用索引查找 | 外表小、内表大且有索引 |
| 块嵌套循环连接 | 按块读取内表 | 较小表作为外表 |
| 归并连接 Sort-Merge Join | 先排序再顺序合并 | 等值连接、输入已排序 |
| 哈希连接 Hash Join | 小表建哈希表，大表探测 | 等值连接、大表无索引 |

### 复习重点（结合考纲）
- ⭐⭐ **常考**：三种连接算法实现策略及适用场景。
- ⭐⭐ **常考**：RBO 与 CBO、选择下推、索引选择。
- ⭐ 了解：执行计划阅读、访问方式。
